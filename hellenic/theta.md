# THETA Protocol: Validation and Quality Assurance

## Purpose

THETA validates completed work. After GAMMA execution, THETA verifies that outputs meet acceptance criteria, that evidence is solid, and that nothing was missed.

**Critical Rule: Do not validate your own assumptions. Validate artifacts.**

## When to Use

- Immediately after GAMMA execution completes
- Before promoting work to next level
- Before committing truth based on evidence
- Before accepting a handoff from another agent

## When NOT to Use

- GAMMA hasn't finished yet
- No clear acceptance criteria exist (run ALPHA first)

## Input Contract

- GAMMA execution report
- Artifacts to validate
- Acceptance criteria
- Validation checklist

## Output Contract

- Validation report with pass/fail for each criterion
- Issues found and recommended fixes
- Sign-off (PASS, REVISE, REJECT) with justification
- Evidence of validation

## Required Evidence

- Checklist of all acceptance criteria
- For each criterion: evidence that it was checked
- Screenshots or logs if applicable
- Clear statement: "I did X to verify Y"

## Forbidden Actions

- Rubber-stamp approval
- Skip tests
- Assume success without checking
- Validate your own claims

## Final Report Format

**THETA Validation Report: [Artifact Name]**

1. **Acceptance Criteria** - Numbered list
2. **Validation Performed** - For each criterion: what check was done, evidence attached
3. **Issues Found** - Numbered list with severity
4. **Recommendations** - What to fix (if issues found)
5. **Sign-Off Decision** - PASS / REVISE / REJECT with brief justification
6. **Validator Notes** - What was assumed, what was tested
